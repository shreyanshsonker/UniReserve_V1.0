1. Introduction & Project Overview
1.1 Purpose
UniReserve is a university facility reservation platform designed to eliminate the chaos of unmanaged academic spaces. Students book study rooms, library seats, computer labs, and specialized facilities in advance. The system enforces booking rules, prevents overcrowding, manages time slots intelligently, and surfaces smart recommendations powered by usage analytics.

This document defines the complete product requirements: database schema, backend architecture, business logic, API surface, and frontend expectations. It is the single source of truth for the engineering team.
1.2 Problem Statement
	•	Students arrive at libraries and labs to find all seats occupied, with no visibility into availability.
	•	Overcrowding in computer labs and study rooms reduces academic productivity.
	•	Facility managers have no data on peak usage, wasted slots, or no-shows.
	•	Administrators cannot enforce fair-use policies without a centralized system.
	•	There is no waitlist mechanism — students who miss out simply lose access to the resource.

2.2 Registration Flows
2.2.1 Student Registration Flow
	•	Student submits form: Name, University Email, Student ID, Department, Year, Password.
	•	Account created with is_active=False, role=STUDENT.
	•	Verification token generated and stored in EmailVerification table.
	•	Async Celery task sends verification email with tokenized link.
	•	Student clicks link → token validated → is_active=True.
	•	Student can now log in and receive JWT tokens.

2.2.2 Facility Manager Registration Flow
	•	Faculty submits form: Name, University Email, Employee ID, Department, Facility they manage.
	•	Account created with is_active=False, role=MANAGER, status=PENDING.
	•	Super Admin sees pending request in dashboard.
	•	Admin approves → is_active=True, approval email sent. Admin rejects → rejection email sent.
	•	Manager can now log in and access their facility dashboard.

2.2.3 Login Flow (All Roles)
	•	User submits email + password to POST /api/auth/login/.
	•	Django validates credentials. If inactive → return specific error.
	•	On success → return access token (15 min) + refresh token (7 days).
	•	Frontend stores tokens in memory (access) and httpOnly cookie (refresh).
	•	Each subsequent request includes Authorization: Bearer <access_token>.
	•	Role extracted from token payload → frontend redirects to role-specific dashboard.

3. Database Schema
3.1 Schema Design Philosophy
The schema follows normalization to 3NF with deliberate denormalization in analytics-heavy tables for query performance. All primary keys are UUIDs except where noted. Soft deletes (is_deleted flag) are used throughout so booking history is never lost. All timestamps are UTC.

MySQL 8.0+ is required for UUID functions, JSON columns, and window functions used in analytics queries.

Slot Generation Logic:
When a manager calls generate-slots, the system creates FacilitySlot rows for each day in the requested range. The slot_duration_mins on the Facility determines the granularity. For example, a facility with operating hours 08:00–20:00 and slot_duration_mins=60 generates 12 slot rows per day. The system skips days marked as blocked or outside operating days. This is done as a background Celery task for large date ranges.

Booking Engine — Core Business Logic:
The booking creation path implements multiple validation layers before committing to the database. Failure at any layer returns a descriptive error response.

	•	Pre-flight checks: Is the student active? Is the student banned? If banned and ban_until is set, has it expired?
	•	Slot validation: Does the slot exist? Is it in the future? Is it blocked by manager? Is the facility active?
	•	Conflict check: SELECT COUNT(*) FROM bookings WHERE student_id=X AND slot_id=Y AND status NOT IN (CANCELLED) — must return 0.
	•	Rule enforcement: Check MAX_BOOKINGS_PER_DAY (count bookings by student for slot_date), check MAX_HOURS_PER_WEEK (sum booking durations for current week), check MIN_ADVANCE_BOOKING_MINS (slot start - now > threshold).
	•	Group size validation: If facility.min_group_size > 1, validate group_size and group_members array.
	•	Capacity check (atomic): SELECT available_capacity - booked_count FROM facility_slots WHERE id=Y FOR UPDATE. If result <= 0 → slot full → offer waitlist.
	•	Write: INSERT into bookings, UPDATE facility_slots.booked_count += 1 in same transaction.
	•	Post-create: dispatch Celery task to send confirmation email and create Notification record.

The FOR UPDATE lock in step 6 is critical. Without row-level locking, concurrent booking requests can both pass the capacity check and both succeed, creating overbooking. This is the race condition that kills most student-built booking systems.

Waitlist Promotion Logic:
When a booking is cancelled, a Django signal (post_save on Booking) triggers a Celery task:
	•	Fetch the cancelled booking's slot_id.
	•	UPDATE facility_slots SET booked_count = booked_count - 1 WHERE id = slot_id.
	•	Query waitlist WHERE slot_id = X AND status = WAITING ORDER BY position ASC LIMIT 1.
	•	If a waitlist entry exists, create a new Booking for that student (same atomic capacity check applies).
	•	Update waitlist entry: status = PROMOTED, promoted_at = NOW().
	•	Dispatch notification email to promoted student. They have 30 minutes to confirm (optional enhancement).

No-Show Detection Logic (Celery Beat scheduled task):
Runs every 5 minutes. For each confirmed booking where slot start time has passed by more than NO_SHOW_CHECKIN_GRACE_MINS and checked_in_at IS NULL:
	•	Update booking.status = NO_SHOW.
	•	Increment user.no_show_count.
	•	If no_show_count >= NO_SHOW_BAN_THRESHOLD: set is_banned = True, ban_until = NOW() + 7 days.
	•	Create Notification and send ban/warning email.
	•	Trigger waitlist promotion for the freed slot.

Recommendation Engine Logic:
Called when a student loads their dashboard. Returns up to 5 slot suggestions.

	•	Fetch student's last 30 bookings → extract facility_id frequency → top 3 preferred facilities.
	•	For each preferred facility, get slots for the next 7 days.
	•	Score each slot: score = (1 - booked_count/available_capacity) × 100 (lower occupancy = higher score).
	•	Boost score by +20 if slot falls within student's historical preferred hours (computed from past bookings).
	•	Filter out slots the student has already booked.
	•	Sort by score descending, return top 5 across all facilities.
	•	If student has fewer than 5 bookings, fall back to globally least-crowded slots for the next 48 hours.

{
  "success": true,
  "data": { ... },
  "message": "Booking confirmed successfully.",
  "meta": { "page": 1, "total": 42, "per_page": 20 }

Error responses:

{
  "success": false,
  "error": {
    "code": "SLOT_FULL",
    "message": "This slot is at full capacity. Would you like to join the waitlist?",

5.3 Authentication
The system uses JSON Web Tokens via djangorestframework-simplejwt. Access tokens expire in 15 minutes. Refresh tokens expire in 7 days and are blacklisted on logout. The token payload includes: user_id, role, email, is_banned, exp. The role field allows the frontend to render the correct dashboard without an extra API call. The is_banned field allows the frontend to gracefully display ban status immediately.

7.2 Facility-Specific Rule Overrides
If a row exists in booking_rules with facility_id set, it overrides the global rule for that specific facility. Example use cases:
	•	Computer labs: MAX_BOOKINGS_PER_DAY = 1 (only one lab session per day per student).
	•	Seminar halls: MIN_ADVANCE_BOOKING_MINS = 1440 (must book at least 24 hours in advance).
	•	Music practice rooms: MAX_HOURS_PER_WEEK = 4 (limited instrument access time).
	•	3D Printing lab: requires_approval = TRUE, MIN_ADVANCE_BOOKING_MINS = 2880 (48 hours notice for equipment prep).

7.3 Group Booking Rules
	•	Facilities with min_group_size > 1 require group_members array in the booking request.
	•	Each member in the group must be a valid active student.
	•	The booking counts toward the quota of ALL members in the group (not just the requester).
	•	If any group member has reached their daily limit, the booking is rejected.
	•	Cancellation by the requester cancels the entire group booking.
