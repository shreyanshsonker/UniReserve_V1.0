1. System Overview
1.1 Architecture Pattern
UniReserve follows a decoupled client-server architecture. The Django REST API backend and React frontend are completely separated — they communicate exclusively through HTTP/JSON over a versioned REST API. This separation allows independent deployment, independent testing, and clean role boundary enforcement at the API layer.

1.2 High-Level Data Flow
	•	React app sends HTTP request with JWT Bearer token in Authorization header.
	•	Django receives request → JWT middleware validates token → extracts user_id and role.
	•	DRF permission class checks role against endpoint requirements. Fails → 403.
	•	View/ViewSet calls a service function (never puts logic in the view directly).
	•	Service validates business rules → queries MySQL via Django ORM.
	•	If async work needed (email, analytics), service dispatches a Celery task.
	•	Serializer converts queryset to JSON → response returned to React.
	•	Celery worker picks up task from Redis queue → executes independently of request cycle.

class UUIDModel(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    class Meta: abstract = True

class TimeStampedModel(UUIDModel):
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
    is_deleted = models.BooleanField(default=False)

    objects = SoftDeleteManager()   # filters is_deleted=False by default
    all_objects = models.Manager()  # unfiltered — for admin and audit queries
    class Meta: abstract = True

// Request: attach access token
api.interceptors.request.use(config => {
  const token = useAuthStore.getState().accessToken;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;

// Response: on 401, attempt token refresh once
api.interceptors.response.use(null, async error => {
  if (error.response?.status === 401 && !error.config._retry) {
    error.config._retry = true;
    const newToken = await refreshAccessToken();
    error.config.headers.Authorization = `Bearer ${newToken}`;
    return api(error.config);
  }
  return Promise.reject(error);
});

8.3 Task: detect_no_shows
Most critical system task. Runs every 5 minutes. Its correctness directly affects student trust and facility utilization.

@shared_task(bind=True, max_retries=3)
def detect_no_shows(self):
    grace = get_rule_value('NO_SHOW_CHECKIN_GRACE_MINS', default=15)
    cutoff = now() - timedelta(minutes=grace)

    # Bookings where slot started > grace_mins ago and no check-in recorded
    late_bookings = Booking.objects.filter(
        status='CONFIRMED',
        checked_in_at__isnull=True,
        slot__slot_date=date.today(),
        slot__start_time__lte=cutoff.time()
    ).select_related('student', 'slot')

    for booking in late_bookings:
        with transaction.atomic():
            booking.status = 'NO_SHOW'
            booking.save(update_fields=['status', 'updated_at'])

            student = booking.student
            student.no_show_count = F('no_show_count') + 1
            student.save(update_fields=['no_show_count'])
            student.refresh_from_db()

            # Check ban threshold
            threshold = get_rule_value('NO_SHOW_BAN_THRESHOLD', default=3)
            if student.no_show_count >= threshold:
                ban_days = get_rule_value('NO_SHOW_BAN_DURATION_DAYS', default=7)
                student.is_banned = True
                student.ban_until = now() + timedelta(days=ban_days)
                student.save(update_fields=['is_banned', 'ban_until'])

        # Async: send email, promote waitlist
        send_no_show_email.delay(booking.id)
        promote_waitlist_on_cancel.delay(booking.slot_id)

9. Security Architecture
9.1 Authentication Flow
Step
Detail
Password hashing
Argon2id (Django's default hasher). Work factor configurable via PASSWORD_HASHERS setting.
JWT access token
HS256, 15-minute lifetime. Payload: {user_id, role, email, is_banned, exp, iat}. Short-lived so role changes propagate quickly.
JWT refresh token
7-day lifetime. Stored as httpOnly cookie (not accessible to JS). Rotation on use — old token blacklisted in DB.
Refresh token blacklist
Django's simplejwt outstanding token table. On logout, refresh token is blacklisted immediately.
Token rotation
Each /token/refresh/ call issues a new refresh token and invalidates the old. Prevents token theft replay.

    def has_permission(self, request, view):

class IsAdmin(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and request.user.role == 'ADMIN'

class IsOwnerOrAdmin(BasePermission):
    def has_object_permission(self, request, view, obj):

class IsNotBanned(BasePermission):
    """Applied on all booking-write endpoints"""
    def has_permission(self, request, view):
        u = request.user
        if u.is_banned:
            if u.ban_until and u.ban_until < now():
                # Auto-unban — ban expired
                u.is_banned = False
                u.save(update_fields=['is_banned'])
                return True
            return False
        return True

11.2 Waitlist Promotion Flow
	•	Student A cancels booking. cancellation_service.cancel() called.
	•	atomic: booking.status = CANCELLED, booking.cancelled_at = now().
	•	atomic: FacilitySlot.booked_count -= 1.
	•	Django signal post_save fires → dispatches promote_waitlist_on_cancel.delay(slot_id).
	•	Celery task: SELECT from waitlist WHERE slot_id=X AND status=WAITING ORDER BY position ASC LIMIT 1.
	•	If entry found: call booking_service.create() for that student (runs full validation pipeline again).
	•	If booking succeeds: update waitlist entry status = PROMOTED, promoted_at = now().
	•	Dispatch send_waitlist_promotion_email.delay(student_id, booking_id).
	•	Re-number remaining waitlist positions: UPDATE waitlist SET position = position - 1 WHERE slot_id=X AND status=WAITING.

python -m venv venv && source venv/bin/activate
pip install -r requirements/development.txt

# 2. Start MySQL and Redis via Docker
docker-compose up -d db redis

# 3. Run migrations and seed data
python manage.py migrate
python manage.py runscript seed_data

# 4. Create superuser
python manage.py createsuperuser

# 5. Start Django
python manage.py runserver

# 6. Start Celery worker (separate terminal)
celery -A unireserve worker -l info -Q emails,system,bookings,analytics

# 7. Start Celery beat (separate terminal)
celery -A unireserve beat -l info --scheduler django_celery_beat.schedulers:DatabaseScheduler

# 8. Start React frontend
cd ../unireserve-frontend && npm install && npm run dev

13.3 Production Checklist
	•	Set DEBUG=False, DJANGO_SETTINGS_MODULE=unireserve.settings.production
	•	Generate strong SECRET_KEY (50+ random chars)
	•	Set ALLOWED_HOSTS to actual domain
	•	Configure CORS_ALLOWED_ORIGINS to frontend domain only
	•	Use Gunicorn with at least 4 workers: gunicorn unireserve.wsgi -w 4 -b 0.0.0.0:8000
	•	Run collectstatic and serve static files via Nginx / CDN
	•	Enable database connection pooling (django-db-geventpool or PgBouncer equivalent for MySQL)
	•	Set up Sentry for error tracking in both Django and React
	•	Configure Celery worker with --concurrency=4 (match CPU cores)
	•	Enable MySQL slow query log for query performance monitoring
